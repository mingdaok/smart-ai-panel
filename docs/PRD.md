# AI Panel Studio - 产品需求文档 (PRD)

## 1. 项目背景与产品定位
**产品定位**：一款“AI圆桌讨论”本地 Web 应用。
**核心价值**：让任何人都能瞬间召集一众“虚拟智囊”，围绕任意话题展开深度碰撞。通过模拟真实的演播厅讨论环境（包括主持人的引导、专家间的自主发言与争论），持续提炼共识与分歧，最终为用户提供多视角的深度分析，打破单一 AI 问答的局限。

## 2. 目标用户与痛点分析
### 目标用户
*   **知识工作者与研究人员**：需要快速对某一议题进行多角度审视。
*   **产品/决策人员**：在方案评审或战略讨论前，希望预演各方立场与潜在争议。
*   **教育与内容创作者**：需要将枯燥知识转化为生动辩论，寻求灵感。

### 用户痛点
1.  **视角单一**：现有的单体 AI 对话往往只提供单一中立或预设立场，缺乏“对抗性思考”。
2.  **异步割裂**：多次单点询问不同角色的 AI，缺乏上下文串联，无法产生真实的碰撞。
3.  **过程不可见**：AI 直接给出总结结论，缺乏思考路径的实时展现。
4.  **议题管理混乱**：用户希望同时探索多个议题，但不同讨论的上下文、进展与结论难以并行隔离管理。

## 3. 核心功能需求 (MVP 阶段)

### 3.1 首页 (Home)
*   **讨论列表展示**：展示所有当前系统内进行中或已结束的讨论列表。
*   **功能入口**：用户可加入观察已有讨论，也可发起新讨论。

### 3.2 嘉宾生成 (Guest Generation)
*   **大模型生成**：用户输入话题与专家人数后，系统自动调用大模型，动态生成一个“主持人 + 专家阵容”。
*   **信息展示**：展示专家的姓名、职业/Title、核心立场，并分配专属的颜色标识。
*   **确认机制**：由用户确认阵容无误后，点击进入沉浸式演播厅。

### 3.3 演播厅模式 (Studio Mode) - 核心交互
*   **主持人机制**：主持人负责整个讨论的生命周期，包括开场白、追问、串联各方观点和最终收尾。
*   **自主发言机制（非机械轮流）**：专家不应按固定顺序发言。后端需实现调度逻辑，专家可根据当前上下文“举手、抢答、补充、反驳”。每次发言控制在 1-2 句核心内容。

### 3.4 专家状态小窗 (Expert Status Window)
*   **独立区域**：界面需为每位参会专家分配独立的小窗。
*   **实时状态反馈**：通过视觉效果实时显示该 Agent 当前处于【待机】、【准备发言】还是【发言中】的状态。
*   **思考摘要**：展示专家当前的关注点或公开思考摘要，但**绝对禁止**展示大模型真实的 chain-of-thought（如隐藏的思维链标签内容）。

### 3.5 实时共识与分歧 (Real-time Consensus & Disagreement)
*   **动态提炼**：系统需在讨论进行中，不断从发言流中提炼当前的“共识”和“分歧”。
*   **实时更新**：这部分内容需在界面指定区域实时动态更新，不等讨论结束就要持续生成并展示。

### 3.6 现场 Transcript (字幕流)
*   **对话流展示**：显示发言人姓名与职业/Title，配合与嘉宾一致的专属色彩区分。
*   **屏蔽内部事件**：Transcript 仅展示可读的交流内容，“举手”、“等待调度”等内部调度事件不予展示。

### 3.7 总结收尾 (Summary)
*   **自然语言总结**：结束时主持人需以自然语言进行总结陈词。
*   **严禁 JSON 暴露**：绝对禁止将 JSON 格式的报文或接口裸数据直接渲染到最终的页面展示中。

## 4. 非功能性需求与技术约束
*   **部署要求**：必须是支持本地运行的前后端分离项目。
*   **数据隔离**：系统必须支持多个讨论并行，不同讨论的状态、事件流、Transcript、共识与分歧必须互相隔离，互不干扰。
*   **实时通信**：强制要求使用 SSE (Server-Sent Events) 或 WebSocket 等实时通信机制来推流。
*   **数据安全**：大模型 API Key 只能在后端从环境变量读取，严禁暴露至前端浏览器。
*   **界面质感**：前端页面为中文 UI，采用响应式布局。不能是整个页面级的滚动，各区域（Transcript、专家区、总结区）必须在自身的容器内独立滚动。超宽屏、普通桌面与窄屏均需合理分配区域。
