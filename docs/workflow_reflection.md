# 开发过程思路与工作流复盘

> **项目名称**：Smart AI Panel (智能多Agent演播厅)
> **技术引擎**：Claude Code + Deepseek V4 Pro

## 1. 自动化流水线：AI 协同开发范式
在本次项目中，我深刻体会到“用 AI 写代码”与“AI 工程化开发”的本质区别。如果只是盲目地把需求丢给大模型，最终往往会得到一堆无法运行、逻辑冲突的“面条代码”。因此，我建立了一条严格的工作流水线：

- **第一步（架构师视角）**：利用大模型极其强大的发散能力，但不允许它写代码，只让它输出 Markdown 格式的架构文档、API 契约和数据库设计（SDD）。
- **第二步（项目经理视角）**：把大文档拆解为 5 分钟可执行完毕的“微任务”（TDD Task），确保上下文窗口极其干净。
- **第三步（码农视角）**：严格按照 TDD 铁律，让 AI 先写失败的测试，再写生产代码。这种“自我验证”机制极大地收敛了大模型的幻觉率。

## 2. 踩坑记录：与 AI 博弈的 3 个经典战役

在协同开发中，我遭遇了几个典型的复合型 Bug，这些 Bug 并非简单的语法错误，而是深入到框架机制与 AI 行为学的层面。

### 战役一：React 严格模式下的“多重影分身”
**现象**：在前端请求大模型生成 8 个专家时，因为 React 18 的 Strict Mode 导致 `useEffect` 挂载两次，结果演播厅里出现了 16 个专家，甚至有重复的名字。
**解决思路**：AI 提出了多种后端去重的方案，但我认为这是典型的治标不治本。最终我引导并要求 AI 回到前端组件生命周期的本质，使用 `useRef` 作为一个防抖并发锁（`generationStartedRef.current`），在 React 组件层完美拦截了双重请求。

### 战役二：调度引擎的“全员闭麦”惨案
**现象**：在集成测试中，所有的专家都一言不发，只有主持人像个相声单口一样连续抛出 6 个问题。
**解决思路**：我排查日志后发现，调度器计算专家发言意愿（score）时，竟然普遍算出了低于 0.3 的分数。通过追溯代码，发现计算立场词汇重叠率 `overlap_ratio` 时，分母错误地使用了 `union(A, B)` 而非 `len(A)`，导致句子越长分数反而越低。修正这行数学公式后，整个演播厅瞬间“活”了过来。

### 战役三：“对牛弹琴”式的语义盲区
**现象**：演播厅跑起来后，虽然发言很激烈，但存在严重的体验硬伤。主持人明确点名了“赵老师”，结果抢到麦的却是“王老师”。
**解决思路**：这是因为原始的调度算法只是纯粹的“关键词重合度”算分，缺乏大模型的“语义理解”。为了不引入沉重的本地 NLP 模型，我在 Python 后端直接实现了一套极简高效的“定向点名雷达”：当检测到上一句话中完整包含某专家的姓名时，赋予极高的权重（`name_bonus = 2.0`），强制实现“点名必起立”的真实交互闭环。

## 3. 对 AI 辅助编程的认知进化

经历了从 0 到 1 的开发，我发现 AI 时代的开发者门槛并没有降低，只是从“编写逻辑”变成了“设计约束”：
1. **AI 犹如一辆没有刹车的跑车**：它生成代码的速度极快，但如果没有 `SDD`（契约约束）和 `TDD`（测试约束），这辆跑车必定会冲下悬崖。
2. **Debug 能力决定了 AI 的上限**：当系统复杂度上升（如本次的 SSE 流式通信、React 闭包、Python 异步调度交织在一起）时，AI 无法“看透”全景。开发者必须具备强大的拆解能力，把复合型 Bug 剥离成具体的单点问题，再给 AI 投喂精准的上下文。
3. **架构决定成败**：如果你要求 AI 直接生成“一个带前后端的网页”，必败无疑。但如果你要求它“写一个 RESTful Controller，入参符合这个 Schema”，它就能完美交差。

总之，这次作业让我完成了从“代码工人”到“AI 牧羊人”的思想转变，这将是我未来工程实践中最宝贵的财富。
