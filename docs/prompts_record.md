# 核心 Prompt 记录文档

本项目严格遵循 AI 辅助开发的工程化范式，通过引导大模型（Deepseek V4 Pro / Claude Code）完成不同阶段的开发任务。以下是核心阶段的代表性 Prompt 及其背后的工程思考。

## 1. [SDD 阶段]：规约与契约生成

**目标**：在不写业务代码的前提下，强制 AI 产出 API 契约和数据库实体图。

**核心 Prompt**：
> "我们现在处于 SDD（Spec-Driven Development）阶段。我要你设计一个『多智能体演播厅』的 MVP。请不要写任何业务代码，仅输出一份完整的 `SDD.md`，必须包含：1) 数据库 SQLite 表结构的 DDL；2) 前后端交互的 REST API 和 SSE 事件流 JSON Schema；3) 演播厅调度的状态机。"

**说明**：
当时为了避免 AI 一上来就写出前后端耦合的混乱代码，我通过这个 Prompt 强制它先进行系统架构设计，明确定义了 `Room`、`Expert`、`Transcript` 等实体关系。如果 AI 试图跳过这一步直接写 React 组件，我会立刻中止并纠正它。

## 2. [DDD 阶段]：前端领域层与 UI 生成

**目标**：基于确定的接口，构建前端视觉呈现和本地状态流转。

**核心 Prompt**：
> "基于我们之前确定的 SSE 事件协议（`room.status`, `expert.state`, `transcript.line`），请使用 React 和 TailwindCSS 构建演播厅（StudioPage）组件。要求：
> 1. 左侧为聊天记录（类似微信对话框，自动滚动到底部）。
> 2. 右侧为专家列表（必须能够实时响应 `expert.state` 的变化，当状态为 preparing 时显示 public_thought）。
> 3. 不要自己 mock 数据，直接对接 `useRoomStore` 的状态。"

**说明**：
前端 UI 生成极易产生“玩具感”。此 Prompt 强调了组件的“数据消费”属性，强制 UI 层只作为后端 SSE 状态流的纯渲染器，确保了前后端数据的绝对一致性，避免了 React 局部状态满天飞的情况。

## 3. [TDD 阶段]：核心逻辑演进与复杂 Bug 修复

**目标**：解决实际业务逻辑中的极值、并发和调度失真问题。

**核心 Prompt**：
> "在我们的 `scheduler.py` 中发现了一个严重的逻辑 Bug。当所有的专家都没有举手时，打分引擎 `_score_experts` 算出的分数因为分子分母倒置（使用了 union 而不是单方 length）导致普遍低于 0.3 的及格线。这导致了主持人连续发言 6 次救场。请分析 `overlap_ratio` 的计算公式，将其从 `overlap / union` 改为 `overlap / len(stance_tokens)`，并重写计算权重的部分。"

**说明**：
在集成测试期间，我们发现多 Agent 讨论陷入了“死寂”（全场只有主持人在问问题）。通过这个深度聚焦的 Prompt，引导 AI 在不破坏原有架构的前提下，精准定位到了算法里的权重分配问题并成功修复。这体现了 AI 辅助调试的威力。

## 4. [E2E 阶段]：系统级质量闭环

**目标**：优化系统级别的真实体验，解决 LLM 在多 Agent 场景下的 NLP 盲区。

**核心 Prompt**：
> "在真实的 E2E 测试中，我们发现一个体验瑕疵：当主持人（LLM）在发言中明确点名『请问赵雅琳老师和刘逸飞老师怎么看』时，紧接着发言的却是没有被点名的王思涵（因为她的立场关键词匹配度更高）。
> 请为 `scheduler.py` 增加一个『NLP 定向点名雷达』：在算分时，检查上一句话 `last_content` 中是否完整包含某位专家的 `name`。如果包含，直接给该专家追加 2.0 的 `name_bonus` 霸王分，强制其抢麦成功。"

**说明**：
这是对整个项目最灵魂的一笔优化。早期的调度器是纯“瞎子摸象”的关键词匹配，这个 Prompt 引导 AI 给调度器加上了“语义感知”能力，使得大模型生成的点名语句能直接在工程调度层生效，大幅提升了演播厅体验的真实感。

## 5. [TDD 阶段附加]：前端框架生命周期防御

**目标**：解决前端 React 并发渲染引起的大模型重复调用灾难。

**核心 Prompt**：
> "在 `LobbyPage` 测试中，我们发现如果开启了 React 18 的 Strict Mode，`useEffect` 会在组件挂载时执行两次。这导致了向后端的 `/experts` 接口连续发送了两次生成请求，最终页面上渲染出了 16 位专家，并且名字存在重复。
> 请不要尝试在后端做复杂的去重拦截，而是直接在前端 `LobbyPage.tsx` 的 `useEffect` 中，引入一个基于 `useRef` 的并发锁 `generationStartedRef`，确保仅发起一次真实请求。"

**说明**：
面对前后端联调时的并发 Bug，AI 经常会给出最容易想到但最难维护的方案（比如在后端加锁）。通过这个 Prompt，我强行把 AI 的视线拉回了“前端生命周期”的领域。这体现了工程化 AI 开发中，人类作为架构师必须具备的“指路”能力，而不是盲目听从 AI 的首选建议。
